# Fleet-command kernel rebuild вЂ” Design

- **Date**: 2026-05-12
- **Status**: Draft (pending user review)
- **Author**: Brainstormed via Claude (Opus 4.7)
- **Scope**: Restructure `fleet-command/` into a "workspace kernel" that bootstraps a portable `cardsense-workspace` across machines, integrating the usable subset of `WaddleStudio/agent-toolkit` and then deleting that repo.

## 1. Background

`fleet-command/` is the single versioned source of truth in `cardsense-workspace/`. The workspace folder itself is not a git repo; sub-repos (`cardsense-*`, project-specific repos) are independently versioned. Local runtime state (`.claude/`, `.gstack/`, `.superpowers/`, `.uv-cache/`) currently lives at the workspace root but is not portable вЂ” moving to a new computer means re-cloning each sub-repo, re-installing each skill, and re-discovering dependency chains.

The user occasionally changes work computers and needs:

- A repeatable setup: `git clone fleet-command + bootstrap` produces a working environment within ~10 minutes.
- Three upstream skills (`superpowers`, `gstack`, `ui-ux-pro-max`) installed reliably without missing dependencies.
- Multi-agent support (Claude Code + Codex CLI at minimum) sharing the same skill set.
- Multiple side projects (CardSense + 6 others) managed from one control plane.
- Clean teardown: a single command removes the workspace with no residue.

`WaddleStudio/agent-toolkit` was created to solve part of this but is metadata-only by design, redundant with `fleet-command`, and its safety philosophy actively blocks auto-install вЂ” opposite to the user's needs. A small subset of its content (uv enforcement policy, cleanup safety pattern, templates) is salvageable.

## 2. Goals

1. Make `fleet-command` the workspace kernel: shipped artifacts (manifest, bootstrap, teardown, configs, templates, recipe skills) fully reproduce the workspace on a new machine.
2. Pin upstream skills by commit SHA (with optional release tag) for reproducibility.
3. CardSense becomes a peer of other side projects under `projects/cardsense/`; no top-level CardSense files.
4. Integrate the salvageable content from `agent-toolkit`, then delete that repo.
5. Migrate via 6 phased PRs so `main` stays buildable throughout.

## 3. Non-goals

- Not sync `~/.claude/`, `~/.codex/`, or any HOME-level state. Auth (`claude login`, `codex login`) stays manual.
- Not auto-`commit`/`push` from bootstrap or teardown.
- Not vendor third-party skill source code into `fleet-command`.
- Not build a cross-agent instruction file generator in this rebuild (deferred); `claude-config/CLAUDE.md` and `codex-config/AGENTS.md` are hand-aligned for now.
- Not implement `adapters/` directory (decided against вЂ” single CLAUDE.md / AGENTS.md source plus per-repo reference is sufficient).

### Boundary rules

- `projects/<name>/` вЂ” content tied to a single side project (spec, status, project-specific docs, dashboard).
- `docs/` вЂ” content that spans multiple projects or is workspace-level (architecture, refs, prompts, policies).
- Heuristic: if a doc describes "shared rules or architecture across projects" в†’ `docs/`; if "internal detail of one project" в†’ `projects/<name>/`.

### 6.3 ref pinning policy

- **Third-party skills**: `ref` must be a structured object with a 40-char `sha`. Optional `tag` for human readability. Optional `resolvedAt` timestamp.
- **Owned repos**: `ref` may be a branch string (e.g. `"main"`) вЂ” intent is to pull latest of owned code on each machine.
- Do not pin a skill to a branch (irreproducible across machines).
- Do not pin a skill to a tag alone (Git tags can be force-moved unless signed/protected).
- Updates flow through `scripts/update-skill-lock.{ps1,sh} --tool <name> --to <tag|sha>`: resolves SHA, updates lock entry, prints unified diff for review, does not auto-commit.

## 7. Bootstrap / teardown behavior

### 7.1 Bootstrap flags

```
bootstrap.{ps1,sh} [flags]
  --install-deps        auto-install missing osDeps (default: check + report)
  --skip-deps           skip osDeps stage
  --skip-skills         skip skill clones
  --skip-repos          skip sub-repo clones
  --skip-agents         skip agent host config deployment
  --update              fast-forward existing repos/skills to manifest ref (default: report drift only)
  --dry-run             print intent, do not execute
  --workspace <path>    override workspace root (default: parent of fleet-command/)
  --verbose
```

### 7.2 Bootstrap stages

| Stage | Reads | Action |
|---|---|---|
| 1. osDeps | `manifest.osDeps.<os>` | Run each `verifyCmd`; missing в†’ print install hint; `--install-deps` в†’ run pkg manager install |
| 2. Repos | `manifest.repos[]` | For each: not present в†’ `git clone --branch <ref>`; present + same ref в†’ skip; present + drift в†’ report (or `--update`) |
| 3. Skills | `skills.lock.json` | `git clone` + `git checkout <ref.sha>`; verify HEAD == ref.sha; run `postClone` if present |
| 4. Agents | `manifest.agents.*` | Copy `templateSource/*` в†’ `target/` per file list; existing file в†’ do not overwrite, print drift hint suggesting `sync-config` |
| 5. Final checks | вЂ” | Print remaining manual steps (`claude login`, token placeholders, etc.) |

### 7.3 Drift / idempotency defaults

- Default: report drift, do not overwrite.
- `--update`: fast-forward to manifest ref.
- Agent host config files (`settings.json`, `CLAUDE.md`, etc.): default never overwrite. Re-syncing template в†’ live requires explicit `sync-config` script (out of scope this rebuild; deferred).

### 7.4 Teardown flags

```
teardown.{ps1,sh} [flags]
  --apply               actually delete (default: dry-run)
  --workspace <path>    absolute workspace path (required if not auto-detected)
  --nuke                also delete fleet-command/ itself (requires all repos clean & pushed)
  --keep <pattern>      exclude pattern (repeatable)
  --verbose
```

### 7.5 Teardown safety guards (integrated from `agent-toolkit/scripts/cleanup-public-pc.sh`)

1. Workspace must be absolute path.
2. Workspace must not be `/`, empty, `$HOME`.
3. Workspace must not be `~/.ssh`, `~/.aws`, `~/.config`, `~/.claude`, `~/.codex`.
4. Workspace must exist and be a directory.
5. Workspace must be non-empty (refuse cleanup-of-nothing).
6. Default mode is dry-run; list intended deletions.
7. `--apply` requires typed `DELETE` to proceed.
8. `--nuke` additional pre-flight: each managed repo must have empty `git status --porcelain` AND empty `git log @{upstream}..`; require typed `NUKE` (different word from `DELETE`).

### 7.6 Teardown target derivation

```
From workspace.manifest.json:
  - repos[].path (excluding fleet-command unless --nuke)
  - agents[].target

From skills.lock.json:
  - skills[].target

Hard-coded cache list:
  - .uv-cache, .gstack, .superpowers, .worktrees, .pytest_cache
```

## 8. Config / templates / skills layer

### 8.1 `claude-config/` (deployed to workspace `.claude/`)

- `settings.json` вЂ” base Claude Code settings: theme, model, `additionalDirectories` allowing sibling sub-repos, `deny` rules for destructive commands, `skills.discoveryPaths`. **No secrets.** Tokens / auth via `claude login` post-bootstrap.
- `CLAUDE.md` вЂ” workspace-wide context: control plane location, sub-repos, tool defaults (uv, Chrome via gstack), recipe skills list, upstream skill routing, branch/commit conventions.

- `config.toml` вЂ” Codex CLI config: `sandbox = "workspace-write"`, `approval = "always"`, model, skill discovery paths.

## 13. Acceptance criteria

After all 6 PRs merge:

1. On a machine that has Git installed but no prior workspace, the sequence `mkdir ws && cd ws && git clone <fleet-command-url> && ./fleet-command/bootstrap/bootstrap.ps1` produces a working workspace: all sub-repos cloned, three skills at pinned SHAs, `.claude/` and `.codex/` deployed. Target end-to-end time: ~10 minutes when OS deps are already present, or ~20 minutes when bootstrap installs them via `--install-deps`.
2. `bootstrap.ps1` rerun is idempotent: prints "no changes" for an already-bootstrapped workspace.
3. `teardown.ps1 --dry-run` lists exactly the intended deletion set; `teardown.ps1 --apply` requires typed `DELETE` and then removes only listed paths.
4. `teardown.ps1 --nuke` refuses if any managed repo has uncommitted or unpushed work; requires typed `NUKE` when clean.
5. `tests/` all pass: `uv run python -m unittest discover tests`.
6. `scripts/render_workspace_assets.py --check` is clean.
7. The `WaddleStudio/agent-toolkit` GitHub repo no longer exists; integration trace is in fleet-command commit history.
